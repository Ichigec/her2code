# Ablation Table Reading Guide

## Common ablation patterns in ML papers

### Block Size Ablation (DFlash-style)

Tests how block size during training vs inference affects acceptance length (tau) and speedup.

Typical table shape (from DFlash paper, Table 5):

| Train BS | Test BS | Math500 tau | Math500 Speedup | HumanEval tau | HumanEval Speedup |
|---|---|---|---|---|---|
| b16 | b16 | 6.33 | 4.64x | 5.29 | 3.96x |
| b16 | b8 | 5.09 | 3.87x | 4.44 | 3.39x |
| b8 | b16 | 5.02 | 3.78x | 4.28 | 3.24x |
| b8 | b8 | 5.21 | 3.97x | 4.61 | 3.53x |

Interpretation:
- **b16 -> b16**: best performance (6.33 tau) — train and test at same block size
- **Asymmetric generalization**: b16-trained works well at b8 (5.09 tau), but b8-trained works poorly at b16 (5.02 tau). Larger blocks = better and more general.
- **b8 -> b8 is worse than b16 -> b16**: 5.21 vs 6.33 — ~18% degradation from smaller block

### Draft Layers Ablation

Tests how many transformer layers in the draft model:

| Layers | Math500 tau | Speedup |
|---|---|---|
| 3 | 5.64 | 4.69x |
| 5 | 5.99 | 4.71x |
| 8 | 6.33 | 4.64x |

Interpretation:
- More layers = higher acceptance (6.33 vs 5.64 for 8 vs 3 layers)
- But speedup peaks at 5 layers because 8 layers adds latency overhead
- 5 layers is the Pareto-optimal point

### Target Features Ablation

| Setting | Math500 tau | HumanEval tau |
|---|---|---|
| 3-H | 5.38 | 4.47 |
| 5-H | 5.64 | 4.61 |

More target hidden features = richer context for draft model. Diminishing returns above 5.

### KV Injection vs Input Fusion

| Variant | GSM8K tau | GSM8K Speedup |
|---|---|---|
| Input Fusion | 4.2 | 2.1x |
| KV Injection | 4.8 | 2.4x |

KV injection is universally better — the draft model sees the actual KV representations, not a projected version.

### Training data ablations

Look for:
- Dataset size vs performance (scaling laws within the paper)
- Data mixture ratios
- Whether training data was generated by target model or comes from external corpora

## General rules for reading ablation tables

1. Each variable should be tested in isolation (one thing changes at a time)
2. The paper's chosen configuration should be the best or near-best in every relevant metric
3. If they chose something that's NOT the best in any column, there should be a non-obvious reason (latency, VRAM, training cost)
4. Numbers that are ~equal (4.64x vs 4.71x) mean the variable doesn't matter much
5. Look for "tau" (acceptance length) as the primary quality metric; speedup is derived from tau
6. Pay attention to test set mismatch (train on A, eval on B) — the drop tells you about generalization
7. Missing values (e.g., no b32/b48/b64 in DFlash ablation) mean the authors never tried those — not that they don't work
