Generated by Knowledge Curator observer agent. Use this as the authoritative architectural reference; the inline sections below are quick-reference summaries.

Hermes does NOT use LangGraph or any DAG framework. The agent is a **simple `while` loop**:

```
run_conversation(agent, user_message):
    messages = [system_prompt, user_msg]
    while budget > 0:
        response = LLM(messages, tools=[...])
        if tool_calls:
            for each: execute tool → append result to messages
            continue
        else:
            return response.text
```

### Three Layers of Agent Creation — All In-Process

Hermes has **three independent paths** that create `AIAgent` instances, and **none of them calls the others**:

```
┌──────────────────────────────────────────────────────────────────┐
│                    ОДИН Python-процесс                           │
│                                                                  │
│  tools/delegate_tool.py          tui_gateway/server.py           │
│    _build_child_agent()            _make_agent()                 │
│      ↓                              ↓                            │
│  AIAgent(model=...)              AIAgent(model=...)              │
│  (subagent delegation)           (desktop/TUI session)           │
│                                                                  │
│  gateway/platforms/api_server.py                                 │
│    _create_agent()                                               │
│      ↓                                                           │
│  AIAgent(model=...)                                              │
│  (HTTP external clients)                                         │
└──────────────────────────────────────────────────────────────────┘
```

| Path | Entry Point | Used By | Mechanism |
|------|-------------|---------|-----------|
| **Delegation** | `tools/delegate_tool.py:_build_child_agent()` | `delegate_task()` tool | Direct `AIAgent()` constructor + `ThreadPoolExecutor.run_conversation` |
| **TUI Gateway** | `tui_gateway/server.py:_make_agent()` | Desktop app (Electron, JSON-RPC) | Direct `AIAgent()` constructor + session dict |
| **API Server** | `gateway/platforms/api_server.py:_create_agent()` | External HTTP clients (PWA, curl, mobile) | Direct `AIAgent()` constructor + `loop.run_in_executor` |

All three call `AIAgent(model=..., ...)` directly — no HTTP between them, no IPC,
no subprocess. The API server is NOT wired to delegation: `delegate_task` still
spawns children in `ThreadPoolExecutor` threads rather than making HTTP calls to
the already-running gateway. For the full implications and what would be needed
to bridge them, see `references/subagent-process-model.md` ("What Hermes Already
Has" and "What blocks it today").

## TUI Gateway (JSON-RPC) Architecture

The `tui_gateway/server.py` is the bridge between the Electron desktop app (or Ink TUI) and the Python backend. It's a **JSON-RPC 2.0 server** that communicates over **stdio or WebSocket** — NOT HTTP.

### Data Flow

```
Electron App (React/TypeScript)
    │  JSON-RPC over stdio or WebSocket
    ▼
tui_gateway/server.py
    │
    ├─ @method("session.create")  → _sessions[sid] dict (lightweight, agent=None)
    ├─ @method("prompt.submit")   → _make_agent() → AIAgent().run_conversation()
    ├─ @method("agents.activate") → agent preset switch (persisted to agent_overrides.json)
    ├─ @method("session.list")    → active sessions
    ├─ @method("session.resume")  → load from state.db
    └─ @method("session.delete")  → cleanup
```

**Key design decisions:**

1. **Deferred agent build.** `session.create` returns immediately with `agent=None` and spawns a background thread that calls `_start_agent_build(sid, session)` → `_make_agent()` → `AIAgent(...)`. The UI renders the composer immediately without waiting for MCP discovery or tool loading. `prompt.submit` calls `_wait_agent()` (blocking on `threading.Event`) if the agent isn't ready yet.

2. **Direct `AIAgent()` call.** `_make_agent()` (line 2629) instantiates `AIAgent(model=..., provider=..., ...)` directly — same as `_build_child_agent()` in delegation. No HTTP, no IPC, no subprocess.

3. **`prompt.submit` blocks the RPC thread.** When the user sends a message, `_run_prompt_submit()` calls `agent.run_conversation(message, stream_callback=_stream)` which blocks the JSON-RPC thread for the entire conversation turn. The UI receives streaming deltas via `_stream` → `message.delta` events, but the RPC method only returns after the agent finishes.

4. **Long handlers go to a thread pool.** Methods like `cli.exec`, `shell.exec`, `session.resume` are routed to `_pool` (4 threads by default) so they don't block the fast path (`approval.respond`, `session.interrupt`).

5. **No connection to `api_server.py`.** The TUI gateway does NOT use the HTTP API server (`:8642`) for anything. It creates its own `AIAgent` instances independently. The API server is a completely parallel path for external clients.

### Session Lifecycle

```
session.create  ──→  _sessions[sid] = {agent: None, ready: Event(), ...}
                         │
                         └─ background: _start_agent_build(sid, session)
                                           │
                                           └─ _make_agent() → AIAgent(model=...)
                                                                │
prompt.submit ──→  _wait_agent(session)  ←─ ready.wait()
                         │
                         └─ agent.run_conversation(text, stream_callback=...)
                              │
                              ├─ SSE: message.delta events to UI
                              ├─ SSE: subagent.progress for delegate_task
                              └─ SSE: message.done at completion
```

## Subagent Lifecycle

```python
# delegate_task() spawns children:
child = _build_child_agent(goal, context, toolsets, model, role, ...)
# Leaf = no delegate_task, orchestrator = can spawn further (bounded by max_spawn_depth)

# Children run in ThreadPoolExecutor:
with ThreadPoolExecutor(max_workers=3) as executor:
    for child in children:
        future = executor.submit(_run_single_child, i, goal, child, parent_agent)

# _run_single_child calls child.run_conversation(goal, system_prompt)
# then extracts summary, registers/unregisters, cleans up
```

### Process Model: Threads, Not Processes

This is a common point of confusion. **Hermes subagents are NOT separate OS processes.** They are `AIAgent` Python objects instantiated in the **same process** as the parent, running in `ThreadPoolExecutor` workers:

| What the user sees | What it actually is |
|---|---|
| `hermes` CLI running → another `hermes` process appears | Separate independent sessions (CLI + desktop + gateway), not delegation |
| `delegate_task` spawns a "worker" | In-process `AIAgent` Python object in a thread |
| Observer system `subprocess.Popen` | Fire-and-forget analytics worker (`observer_worker.py`) — NOT delegation |
| Batch (N tasks) | `ThreadPoolExecutor(max_workers=N)` — still same process |

**Parent blocks waiting for children** (synchronous model):
- Single child: `_child_future.result(timeout=child_timeout)` — blocks
- Batch: polling loop `wait(pending, timeout=0.5, return_when=FIRST_COMPLETED)` — blocks

The parent cannot do other work while children run. Each child runs its own `run_conversation()` while-loop (same code as the parent), but they share the Python GIL. Since most subagent time is spent waiting for HTTP responses from the LLM provider, the GIL is not a bottleneck in practice.

**Resource implications:**
- Each child = 1 Python thread (~8MB stack + TLS overhead per thread)
- Threads share heap memory with parent (credential pool, file state, callbacks)
- Context switching is OS-level (compared to Node.js event loop which is user-level)
- For a comparison with OpenCode's async-HTTP session model (which avoids per-child thread overhead), see `references/subagent-process-model.md`.
- **The API server bridge**: Hermes already has an HTTP-based agent lifecycle at `gateway/platforms/api_server.py` (`_create_agent`, `_run_agent`, `/v1/runs`) that could replace in-process threads for delegation — but it's not wired to `delegate_task` yet. See "What Hermes Already Has" in `references/subagent-process-model.md` for the full analysis and what blocks it.

Blocked tools for children: `delegate_task`, `clarify`, `memory`, `send_message`, `execute_code`.

## Desktop Build & Deploy Verification

When modifying the desktop Electron app (`apps/desktop/`), changes go through
three layers that must all be consistent:

1. **Source** (`apps/desktop/src/`) — TypeScript/React source files
2. **Build output** (`apps/desktop/dist/`) — Vite-bundled JS (`npm run build`)
3. **Packaged app** (`apps/desktop/release/linux-arm64-unpacked/resources/app.asar`) — Electron-builder asar archive

`hermes gui` auto-rebuilds when `desktop-build-stamp.json` (content hash of

## Gateway Slash-Command Dispatch

Slash commands need handlers in **two places** to work everywhere:

| Layer | File | Purpose |
|-------|------|---------|
| Command definition | `hermes_cli/commands.py` | Registers the command name, aliases, subcommands |
| CLI handler | `cli.py` | `_handle_*_command()` — used by interactive CLI (`hermes`) |
| Gateway handler | `gateway/run.py` | `_handle_*_command()` — used by desktop app, Telegram, Discord, etc. |
| Gateway dispatch | `gateway/run.py` ~line 8200-8440 | Chain of `if canonical == "..."` that routes to handlers |

The desktop app and all messaging platforms go through the gateway, **not** the CLI. A command registered in `commands.py` without a gateway handler will be silently forwarded to the LLM as plain text — the user sees no error, but the command doesn't take effect.

### Adding a gateway handler for a new slash command

1. Add `if canonical == "yourcmd": return await self._handle_yourcmd_command(event)` to the dispatch chain in `gateway/run.py` (insert alphabetically among existing handlers)
2. Implement `async def _handle_yourcmd_command(self, event)` — follow the pattern of existing handlers like `_handle_model_command` or `_handle_reasoning_command`
3. The command will appear in `GATEWAY_KNOWN_COMMANDS` automatically if `cli_only` is not set on its `CommandDef`

## Pitfalls

- **Code exploration**: use `read_file` and `search_files` directly. Do NOT use `execute_code` for file reading — it triggers consent prompts that stall and frustrate the user. `execute_code` is for multi-step processing scripts (3+ tool calls with logic between them), not for `grep`/`cat` equivalents.
- **CRITICAL: `provider:` in agent `.md` frontmatter was SILENTLY IGNORED — FIXED 2026-07-10.** `AgentDef` had no `provider` field, `_FRONTMATTER_FIELDS` whitelist omitted `"provider"`, and `apply_agent()` called `switch_model(new_model=agent_def.model)` WITHOUT the required positional `new_provider` — causing `TypeError: missing 1 required positional argument: 'new_provider'`, silently swallowed by `except Exception`. Net effect: `/agent plan3` stayed on the session-default model+provider regardless of frontmatter. **Fix applied (4 patches to `agent/agents.py`):** (1) Added `provider: Optional[str] = None` to `AgentDef`, (2) Added `"provider"` to `_FRONTMATTER_FIELDS`, (3) `_coerce_agent_def` reads `data.get("provider")`, (4) `apply_agent` now calls `switch_model(new_model=..., new_provider=agent_def.provider or agent_obj.provider)`. After fix, `/agent plan3` with `provider: custom:local` correctly switches both model AND provider. Full verification: `references/agent-provider-field-fix.md`. Subagent routing via `delegate_task(provider=...)`` in the instruction body still matters — but now the parent session's provider is also correct (the `.md` prose that tells the LLM what arguments to pass). To verify delegate_task calls will resolve:
  ```python
  from hermes_cli.runtime_provider import resolve_runtime_provider
  resolve_runtime_provider(requested='custom:kimi', target_model='kimi-k2.7-code')
  ```
  If it raises `Unknown provider`, either add a `custom_providers:` block to `~/.hermes/config.yaml` or replace `custom:kimi` with the built-in `kimi-coding` in the `.md` delegate_task instructions. Note: even with `custom_providers` configured, `/agent <preset>` will NOT switch the provider — only `delegate_task(provider=...)` uses it.
- **CRITICAL: read_file → write_file API mismatch.** `read_file` returns content prefixed with line numbers (`"1|import foo\n2|..."`). NEVER pass this content directly to `write_file` — the file will be written with literal `"1|"` prefixes, corrupting it. For modifications: use `patch` tool (exact text replacements).
