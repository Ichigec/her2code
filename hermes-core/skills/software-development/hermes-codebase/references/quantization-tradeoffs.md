# Model Selection: Quantization Tradeoffs

Ключевой принцип: **лучше модель поменьше в хорошей квантизации, чем гигант в плохой.**

## Правило

```
35B Q8_0 (74.4 SWE-Bench, ~100% качества) > 397B IQ2_XS (~50 SWE-Bench, ~90% качества)
```

Квантизация убивает преимущество размера. Big model + bad quant < smaller model + good quant.

## Деградация по уровням (Llama-3.1-8B, arXiv 2601.14277)

| Квант | Avg vs FP16 | GSM8K (матем.) | Размер | Вердикт |
|---|---|---|---|---|
| Q8_0 | −0.06 | 77.5 | 53% | ✅ Эталон |
| Q6_K | −0.24 | 78.2 | 41% | ✅ Отлично |
| Q4_K_M | −0.32 | 77.4 | 31% | ✅ Хорошо |
| Q3_K_L | −0.69 | 74.1 | 27% | ⚠️ Приемлемо |
| Q3_K_M | −1.40 | 73.2 | 25% | ⚠️ Заметно |
| Q3_K_S | −3.98 | 68.3 | 23% | 🔴 Плохо |
| IQ2_XS | ~−5+ | — | ~17% | 🔴 Очень плохо |

## Что чувствительно к квантизации

- **GSM8K (математика):** −9.3 на Q3_K_S — максимальная деградация
- **IFEval (инструкции):** −5.0 на Q3_K_S
- **MMLU (знания):** −4.2 на Q3_K_S
- **HellaSwag (здравый смысл):** −0.6 — почти нечувствителен

## MoE модели

MoE (35B/3B) более чувствительны к квантизации чем dense модели того же активного размера (3B), но устойчивее чем dense того же тотального размера (35B). Квантизация сжимает ВСЕХ экспертов, включая неактивных.

## Пример: Nex-N2-Pro (397B/17B) на DGX Spark 128GB

```
Nex-N2-Pro IQ2_XS:  ~120 GB, ~50 SWE-Bench (оценка после деградации)
Nex-N2-mini Q8_0:    ~35 GB, 74.4 SWE-Bench

Вывод: Nex-N2-mini Q8_0 лучше для coding.
```

## Практическое правило для DGX Spark

1. Основная модель: ≤35B в Q8_0 (35GB) — эталонное качество
2. Запасная: ≤35B в Q4_K_M (20GB) — 99.5% качества
3. Никогда: >100B в IQ2_XS — деградация съедает преимущество размера
